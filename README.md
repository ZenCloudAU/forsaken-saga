# The Forsaken Saga

The Forsaken Saga is a retro-inspired narrative RPG and worldbuilding project built in public. It combines old-school CRPG design, interactive storytelling, visible dice systems, character-driven fantasy writing, and modern AI-assisted development workflows.

This repository is the source of truth for the game, world bible, books, devlog, studio operating model, learning record, tool register, command log, public GitHub Pages site, and sprint delivery artefacts.

## Studio Principle

Phil codes and learns the craft. The project copilot handles product management, design support, UX framing, documentation, release discipline, tooling records, command tracking, devlog structure, sprint control, and administrative execution.

The operating rule is simple: capture everything, implement selectively.

## Current Sprint

Sprint 00 — Studio foundation.

The goal is not to build game mechanics yet. The goal is to establish the project brain: repository structure, operating model, project brief, roadmap, architecture notes, tool register, command log, Obsidian-style vault, devlog, and public web shell.

## First Playable Direction

The first playable prototype will be deliberately small: one room, one character, one NPC, one visible internal-state mechanic, one dialogue choice, one consequence, one devlog entry.

## Technology Direction

The initial game engine is Godot 4 using GDScript. Markdown is used for the learning journal, world bible, devlog, command log, decisions, and public documentation. GitHub is the source-control and publication spine.

## Repository Map

`game/` contains the Godot project when created.

`world-bible/` contains canon, lore, factions, characters, locations, timelines, and design pillars.

`books/` contains manuscript material and narrative development.

`devlog/` contains public build diary entries.

`docs/` contains the GitHub Pages website.

`assets/` contains screenshots, concept art, UI references, sprites, and media.

`studio/` contains the operating model, roles, rituals, and delivery rules.

`context/` contains project context for AI-assisted work.

`tasks/` contains sprint plans, backlog, bugs, and future work.

`decisions/` contains architecture decision records.

`vault/` contains the Obsidian-style learning and build journal, including tools, commands, and session notes.

## Day 00 Outcome

The project begins with structure, not chaos. Once Sprint 00 is stable, Phil starts coding inside Godot while the supporting system records what was learned, what was built, what commands were run, and why each decision was made.
